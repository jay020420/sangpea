summary: 생성 모델이 RAG를 제품 사실로 오해하지 않고 구조, 톤, 위험 표현, 편집 흐름 보강용으로만 쓰도록 하는 우선순위와 금지 규칙이다.

제품 사실 대체 금지: 이 문서는 RAG 사용 정책만 제공한다. RAG에 있는 일반 지식은 현재 상품의 사실이 아니며, 업로드 자료나 사용자 입력으로 확인될 때만 확정한다.

목적: 생성 모델이 공통 지식 RAG를 어떻게 사용해야 하는지 정의한다.

## 우선순위

1순위는 사용자가 방금 업로드한 제품 이미지, 앱 화면, 웹서비스 스크린샷, 기존 상세페이지 자료와 입력한 요청이다.
2순위는 기존 상세페이지 이미지 또는 PDF에서 확인되는 실제 정보다.
3순위는 공통 RAG 지식이다.
4순위는 일반적인 상세페이지 구성 지식이다.

RAG 지식은 제품 사실을 대체하지 않는다. RAG는 구조, 카피 방향, 리스크 점검, 카테고리별 체크리스트를 보강하는 용도다.

## RAG를 쓰는 방법

제품/SW 분석 단계에서는 RAG에서 카테고리별 구매 기준, SW 도입 기준, 필수 정보, 위험 표현을 찾아 섹션 설계에 반영한다.

이미지 생성 단계에서는 RAG에서 시각 스타일, 금지 표현, 텍스트 오버레이 규칙, SW 화면 보존 규칙을 참고한다.

리디자인 단계에서는 RAG에서 진단 기준을 참고해 기존 페이지의 문제를 분류하고, 원본 정보를 보존하면서 섹션 순서를 재구성한다.

FAQ 생성 단계에서는 RAG에서 카테고리별 구매 불안을 찾아 질문으로 바꾼다.

YouTube 기반 RAG는 상세페이지 제작 방식, 자동화 흐름, 편집/검수 기준을 보강하는 용도로만 쓴다. 영상 제목이나 설명에 있는 도구 홍보 문구, 가격 후킹, 무료 공개 문구를 현재 상품의 장점처럼 쓰지 않는다.

## RAG 사용 시 금지

RAG에 일반적인 예시가 있다고 해서 현재 제품의 사실로 단정하지 않는다.

예를 들어 RAG에 “화장품은 제형과 사용감을 보여준다”가 있어도, 현재 제품이 화장품인지 확인되지 않으면 화장품 전용 섹션을 만들지 않는다.

RAG에 “인증은 근거가 필요하다”가 있다고 해서 인증 배지를 만들지 않는다. 실제 인증 자료가 있어야 한다.

카테고리별 예시는 현재 제품과 맞을 때만 사용한다. 제품 이미지와 사용자 요청에 없는 기능을 추가하지 않는다.

SW/SaaS 예시는 현재 서비스와 맞을 때만 사용한다. 스크린샷이나 사용자 입력에 없는 기능, 요금제, 고객사, 보안 인증, 연동 목록, 성과 수치, 대시보드 데이터는 추가하지 않는다.

영상 기반 RAG에 특정 외부 서비스명, 요금, 혜택이 있어도 생성 결과에 넣지 않는다. 사용자가 명시적으로 그 서비스 비교를 요청한 경우가 아니면 내부 제작 참고로만 처리한다.

## 불확실성 처리

확실하지 않은 정보는 생성하지 말고 다음 중 하나로 처리한다.

“사용자 확인 필요”
“제공 자료 기준 확인 불가”
“제품 표시사항 확인 필요”
“인증/시험성적서 제공 시 반영 가능”

최종 사용자 화면에는 너무 많은 “확인 필요” 문구를 노출하지 않는다. 대신 섹션 설계 내부 메모나 오류 메시지에서 사용자에게 추가 자료를 요청한다.

## 좋은 생성 응답

좋은 응답은 섹션별로 목적이 분명하고, 제품 이미지에서 확인 가능한 장점과 RAG에서 가져온 구조 지식을 구분한다.

SW 홍보용 상세페이지에서는 스크린샷에서 확인되는 UI와 사용자가 입력한 기능만 사실로 다루고, RAG의 SaaS 지식은 데모 흐름, CTA, 도입 불안, 보안/개인정보 FAQ 같은 구조에만 사용한다.

예:
“제품 이미지에서 확인되는 원통형 패키지와 휴대성을 중심으로 Hero를 구성한다. RAG 기준상 첫 화면은 제품 정체와 핵심 구매 이유를 동시에 보여줘야 하므로, 휴대 장면과 제품 클로즈업을 결합한다. 효능 표현은 자료가 없어 사용하지 않는다.”

## 나쁜 생성 응답

나쁜 응답은 RAG의 일반 지식을 제품 사실처럼 말한다.

예:
“이 제품은 KC 인증을 받은 안전한 유아용 제품이며, 임상적으로 검증되었습니다.”

위 문장은 제품 이미지나 사용자 자료에 근거가 없으면 생성하면 안 된다.
