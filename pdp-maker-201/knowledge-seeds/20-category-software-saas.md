summary: 소프트웨어, SaaS, 앱, 웹서비스, 프로그램, AI 툴, B2B 디지털 제품의 홍보용 상세페이지에서 기능보다 사용 맥락, 데모, 업무 흐름, 신뢰, 보안, 요금제, 도입 불안을 다루는 카테고리 전용 지식이다.

제품 사실 대체 금지: 이 문서는 SW/SaaS/앱 상세페이지 구조와 표현 기준만 제공한다. 실제 기능, 화면, 요금제, 고객사, 보안 인증, 성능 수치, 지원 범위, 연동 목록은 사용자가 제공한 자료나 업로드 이미지에서 확인될 때만 쓴다.

목적: 물리 상품이 아닌 소프트웨어와 디지털 서비스도 한국형 홍보용 상세페이지로 만들 수 있도록 구매 기준, 섹션 흐름, 카피 톤, 이미지 연출, 금지 표현을 정의한다.

## 구매자는 무엇을 보는가

SW 구매자는 "무엇을 할 수 있는가"보다 "내 일이 어떻게 쉬워지는가"를 먼저 본다. 대상 사용자, 해결하는 업무, 기존 방식의 불편, 도입 후 흐름, 학습 난이도, 가격 부담, 데이터 안전성, 협업 방식, 지원 범위가 핵심 판단 기준이다.

B2B SaaS라면 담당자 개인의 편의뿐 아니라 팀 도입 명분이 필요하다. 관리자에게는 권한 관리, 리포트, 보안, 연동, 정산, 계약, 교육 자료가 중요하고, 실무자에게는 반복 업무 감소, 오류 감소, 빠른 확인, 쉬운 사용법이 중요하다.

소비자 앱이나 생산성 도구라면 첫 사용까지의 장벽을 낮춰야 한다. 설치 여부, 회원가입 단계, 무료 체험, 예시 템플릿, 첫 결과를 얻는 시간, 모바일/PC 지원, 데이터 백업, 해지/환불 조건이 구매 불안을 줄인다.

## 권장 상세페이지 흐름

Hero: 제품명보다 카테고리와 사용자가 얻는 결과를 먼저 보여준다. 예: "CS 문의를 한 화면에서 정리하는 팀용 헬프데스크"처럼 대상, 업무, 결과가 한 문장에 들어가야 한다.

Problem: 현재 방식의 불편을 구체적으로 짚는다. "엑셀, 메신저, 메일을 오가며 누락되는 승인 요청"처럼 실제 업무 장면으로 말하고, 공포나 과장 대신 공감 가능한 병목을 보여준다.

Demo / How it works: 실제 화면 또는 제공된 스크린샷을 기준으로 3~4단계 사용 흐름을 보여준다. 가입, 연결, 입력, 자동 처리, 결과 확인, 공유 같은 순서가 좋다.

Core benefits: 기능 목록을 그대로 나열하지 않는다. 기능을 "사용자 이득"으로 바꾼다. 예: "API 연동"은 "기존 주문 데이터를 다시 입력하지 않아도 된다"로 연결한다.

Use cases: 역할이나 상황별로 나눈다. 1인 사업자, 마케팅팀, 운영팀, 개발팀, 대표, 고객센터, 강사, 쇼핑몰 운영자처럼 구매자가 자신을 찾을 수 있게 한다.

Proof / Trust: 고객사 로고, 후기, 수치, 보안 인증, 성과 사례는 자료가 있을 때만 쓴다. 자료가 없으면 "보안 설정 확인", "연동 가능 범위 확인", "샘플 화면으로 기능 확인" 같은 검증 가능한 행동으로 대체한다.

Pricing / CTA: 물리 상품의 "구매하기"보다 "무료 체험 시작", "데모 보기", "상담 신청", "요금제 확인", "기능 둘러보기", "설치 가이드 보기"가 자연스럽다. 가격이 제공되지 않았으면 임의 요금표를 만들지 않는다.

FAQ / Risk reversal: 보안, 개인정보, 데이터 이전, 기존 툴 연동, 팀원 초대, 권한 관리, 지원 시간, 해지, 환불, 무료 체험 종료 후 과금 여부를 묻는다. 답은 제공 자료가 있을 때만 확정하고, 없으면 "확인 필요"로 남긴다.

## 이미지 연출 기준

SW 상세페이지의 중심 이미지는 실제 제품 화면, 앱 스크린샷, 웹 대시보드, 브라우저 프레임, 모바일 프레임, 업무 흐름 카드, Before/After 워크플로우, 기능 콜아웃이다.

업로드된 스크린샷이 있으면 UI 구조, 메뉴명, 화면 색상, 실제 텍스트, 차트 형태를 보존한다. 모델이 임의로 새로운 기능 화면, 가짜 고객 데이터, 가짜 매출 차트, 가짜 알림, 가짜 로고, 가짜 보안 배지를 만들면 안 된다.

화면 안의 작은 글자가 깨질 가능성이 높으면 핵심 카피는 편집기 텍스트 오버레이로 둔다. 이미지에는 화면 구조와 시각 흐름을 만들고, 가격, CTA, 기능명, 보안 문구, 요금제명은 수정 가능한 텍스트 레이어로 관리한다.

서비스 화면이 제공되지 않았으면 추상적인 노트북 목업만 크게 쓰지 않는다. 사용자가 제공한 설명을 기준으로 "업무 흐름 다이어그램", "단계 카드", "체크리스트", "역할별 사용 장면"처럼 사실을 덜 왜곡하는 시각화를 선택한다.

## 한국어 카피 톤

한국형 SW 카피는 과한 혁신/자동화/AI 만능 표현보다 "지금 쓰는 방식에서 무엇이 줄어드는지", "팀원이 왜 납득할 수 있는지", "도입 전에 무엇을 확인하면 되는지"를 차분하게 보여주는 편이 좋다.

좋은 문장: "문의, 주문, 정산 내역을 한 화면에서 확인합니다." "처음 쓰는 팀원도 단계별로 따라갈 수 있습니다." "보안과 연동 범위는 도입 전 확인할 수 있습니다."

주의 문장: "매출 10배", "업무 100% 자동화", "국내 1위", "완벽한 보안", "무조건 비용 절감", "모든 툴과 연동", "개인정보 걱정 끝", "AI가 전부 해결"은 근거 자료가 없으면 쓰지 않는다.

## 섹션별 프롬프트 예시

Hero 예시: "제공된 앱 스크린샷의 UI 색상과 메뉴 구조를 보존한다. 모바일 홍보용 상세페이지 Hero 섹션. 브라우저 프레임 안에 실제 대시보드 화면을 배치하고, 상단에는 편집 가능한 한 줄 헤드라인 영역을 둔다. 고객사 로고, 보안 인증, 성과 수치는 만들지 않는다."

Demo 예시: "제공된 화면을 기준으로 3단계 사용 흐름 카드로 구성한다. 1 입력, 2 자동 정리, 3 공유 또는 확인의 흐름을 시각화한다. 실제 화면에 없는 버튼명과 기능명은 만들지 않고, 긴 설명 문단은 넣지 않는다."

Benefit 예시: "기능을 3개 카드로 나누되 각 카드는 고객 이득 중심으로 표현한다. 화면 일부 클로즈업, 아이콘, 짧은 편집 텍스트를 사용한다. 가짜 그래프 상승, 가짜 사용자 후기, 가짜 수치를 넣지 않는다."

FAQ 예시: "보안, 요금제, 무료 체험, 연동, 데이터 이전, 지원 범위를 묻는 FAQ형 섹션을 만든다. 답변은 제공 자료가 없으면 구체 수치를 쓰지 않고 '도입 전 확인' 중심으로 둔다."
