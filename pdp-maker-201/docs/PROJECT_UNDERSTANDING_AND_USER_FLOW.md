# PDP Maker 프로젝트 이해와 유저플로우

## 이 문서의 목적

이 문서는 현재 PDP Maker를 내가 어떻게 이해하고 있는지, 그리고 사용자가 실제로 어떤 흐름으로 작업해야 하는지를 정리한다. 특히 최근 논의에서 중요한 정정이 있었다.

기존의 "카피가 들어갈 빈 공간을 둔 배경 이미지를 생성하고 그 위에 텍스트를 올린다"는 접근은 Canva나 Mangoboard형 에디터의 핵심과 맞지 않는다. 에디터를 붙이는 목적은 이미지 한 장을 꾸미는 것이 아니라, 텍스트, 이미지, 도형, CTA, 증빙 카드, 제품컷을 모두 독립 레이어로 다루는 문서를 만드는 것이다.

따라서 이 프로젝트의 중심 산출물은 생성 이미지가 아니라 편집 가능한 계층형 상세페이지 문서다.

## 프로젝트의 핵심 목표

PDP Maker는 제품 이미지, SW 화면, 기존 상세페이지, PDF, 상품 설명을 바탕으로 한국형 모바일 상세페이지를 빠르게 구성하고 편집하는 로컬 제작 도구다.

목표는 다음과 같다.

- 사용자가 자료를 넣으면 상세페이지 섹션 구조와 카피 초안을 만든다.
- 섹션마다 적절한 문서 템플릿을 선택해 초기 레이아웃을 만든다.
- 이미지는 최종 섹션 전체를 대신 생성하는 것이 아니라, 필요한 시각 자산으로 생성하거나 배치한다.
- 카피, CTA, 증빙, 스펙, FAQ, 도형, 배경, 제품컷은 모두 편집 가능한 레이어로 남긴다.
- 최종 산출물은 JPG/ZIP뿐 아니라 Figma payload, layered document, manifest처럼 재편집 가능한 데이터도 포함한다.

## 핵심 제품 원칙

### 1. 에디터 문서가 source of truth다

상세페이지의 최종 상태는 `LayeredDocumentV2`와 에디터 레이어 상태가 기준이다. 생성 이미지는 그 문서 안에 들어가는 자산 중 하나일 뿐이다.

예를 들어 Hero 섹션은 다음처럼 구성되어야 한다.

- 제품 이미지 또는 SW 화면 레이어
- 헤드라인 텍스트 레이어
- 서브카피 텍스트 레이어
- 신뢰 문장 또는 증빙 칩 레이어
- CTA 버튼 도형 + 텍스트 레이어
- 배경 도형 또는 패턴 레이어

이 구성을 한 장의 이미지로 생성한 뒤 텍스트만 위에 붙이는 방식은 목표가 아니다.

### 2. AI 이미지 생성은 전체 섹션 렌더러가 아니다

AI 이미지 생성의 역할은 다음에 가깝다.

- 제품이 놓인 장면 이미지 생성
- SW 화면을 담는 디바이스 프레임 또는 맥락 이미지 생성
- 배경 텍스처, 패턴, 조명, 소재감 생성
- 특정 섹션에 필요한 보조 비주얼 생성
- 기존 제품컷을 더 좋은 컷처럼 보정하거나 확장

반대로 다음은 지양한다.

- 헤드라인/CTA/스펙까지 포함된 완성 섹션 이미지 생성
- 카피가 들어갈 큰 빈 공간을 전제로 한 배경 plate 생성
- 텍스트 안전영역을 이미지 생성 모델이 알아서 설계하게 하는 방식
- 템플릿 역할을 프롬프트만으로 해결하려는 방식

레이아웃은 에디터와 템플릿 시스템이 담당하고, 이미지 생성은 레이어에 들어갈 시각 자산을 담당해야 한다.

### 3. 템플릿은 프롬프트가 아니라 문서 구조다

`hero-product-focus`, `problem-checklist`, `benefit-card-grid`, `proof-spec-panel`, `demo-step-flow`, `usecase-split-scene`, `faq-final-cta` 같은 템플릿은 이미지 스타일 힌트가 아니라 레이어 구조의 프리셋이어야 한다.

각 템플릿은 다음을 가져야 한다.

- 기본 프레임 크기
- 텍스트 레이어 슬롯
- 이미지/제품컷 슬롯
- 도형/카드/CTA 슬롯
- 기본 z-index
- 모바일 가독성 기준
- 섹션 목적에 맞는 카피 타입

사용자가 템플릿을 바꾸면 이미지 프롬프트만 바뀌는 것이 아니라 레이어 배치와 편집 가능한 구성 요소가 바뀌어야 한다.

### 4. 카피는 이미지 위에 붙이는 보조 요소가 아니다

카피는 상세페이지의 중심 데이터다. 이미지 생성 이후 덧붙이는 텍스트가 아니라, 섹션 목적과 구매 전환 흐름을 만드는 핵심 레이어다.

따라서 카피 워크플로우는 다음처럼 분리되어야 한다.

- 상품 사실과 증빙에서 카피 후보를 만든다.
- 구매자가 각 섹션에서 던지는 질문을 먼저 정한다.
- 각 섹션 카피가 답해야 할 각도, 근거, 금지 표현을 명시한다.
- 각 카피를 headline, subheadline, bullet, proof, CTA, FAQ 등 역할로 구조화한다.
- 템플릿의 텍스트 슬롯에 매핑한다.
- 에디터에서 줄바꿈, 크기, 색상, 배경 도형, 위치를 조정한다.
- 최종 export 시 이미지처럼 합성하되, 내부 데이터는 계속 텍스트 레이어로 유지한다.

카피 생성은 다음 품질 기준을 통과해야 한다.

- 첫 화면은 상품명 또는 제품군과 핵심 사용 맥락을 즉시 드러낸다.
- 문제 섹션은 실제 고객 상황을 짚고 공포 마케팅으로 흐르지 않는다.
- 베네핏 섹션은 기능명이 아니라 고객이 얻는 변화로 번역한다.
- 선택 이유 섹션은 경쟁사 비방 없이 판단 기준을 제시한다.
- 근거 섹션은 업로드 자료에 있는 증빙만 쓴다. 후기, 인증, 순위, 수치, 보장 표현은 근거가 없으면 만들지 않는다.
- CTA는 "자세히 보기" 수준에서 끝나지 않고, 섹션 맥락에 맞는 다음 행동을 말한다.
- 프리미엄, 혁신, 완벽, 압도적 같은 추상어는 근거 있는 사실로 바꾼다.

## 현재 구현에 대한 이해

현재 프로젝트는 이미 다음 방향으로 이동하고 있다.

- `react-konva` 기반 캔버스가 추가되어 레이어 편집을 시작했다.
- 텍스트와 사각형 레이어를 추가, 선택, 복제, 순서 변경, undo/redo 할 수 있다.
- 섹션별 디자인 템플릿 ID가 들어가기 시작했다.
- `LayeredDocumentV2`, Figma payload, export manifest가 만들어진다.
- 생성 이미지가 없어도 템플릿 기반 문서를 먼저 편집할 수 있다.

최근 수정으로 이미지 생성은 배경 plate가 아니라 시각 에셋 생성으로 전환됐다. 남은 핵심 부채는 "좋은 산출물"을 모델이 한 번에 맞히기를 기대하는 것이 아니라, 사용자가 중간 산출물을 검수하고 수정한 뒤 다음 비용 단계로 넘어가도록 만드는 것이다.

- 스토리라인 검수 없이 이미지 생성으로 넘어가면 섹션 목적이 틀어진 상태에서 비용이 큰 생성이 실행된다.
- 이미지 계획 검수 없이 생성하면 제품/화면/증빙 에셋 대신 포스터형 장면이나 가짜 UI가 나올 수 있다.
- 최종 생성물 검수 없이 export가 열리면 미검수 결과가 납품될 수 있다.
- 따라서 에디터는 분석 결과를 곧바로 완성물로 취급하지 않고, `스토리라인 승인`, `이미지 계획 승인`, `최종 검수 승인`을 명시적 게이트로 다뤄야 한다.

## 이상적인 사용자 흐름

### A. 새 PDP 제작 흐름

1. 사용자가 제품 이미지, SW 화면, PDF, 참고 자료를 업로드한다.
2. 사용자가 상품 설명, 금지 표현, 톤, 채널을 입력한다.
3. LLM이 자료를 분석해 제품 사실, 부족한 정보, 사용 가능한 근거, 위험한 주장 후보를 정리한다.
4. LLM이 상세페이지 스토리라인과 카피 전략을 만든다. 이 단계의 산출물은 섹션 순서, 섹션 역할, 구매자 질문, 카피 각도, 헤드라인, 서브카피, 근거, CTA 초안이다.
5. 사용자가 스토리라인과 카피 전략을 검수한다. 수정이 필요하면 섹션 목적, 순서, 카피, 근거, 금지 표현을 수정하고 다시 승인한다.
6. 스토리라인이 승인되면 시스템이 섹션별 이미지 계획을 만든다. 이 단계의 산출물은 필요한 시각 에셋, 참조 자료 사용 방식, 생성 옵션, 템플릿, 최종 프롬프트다.
7. 사용자가 이미지 계획을 검수한다. 수정이 필요하면 실제 제품/화면/증빙 중심으로 계획을 바꾸고 다시 승인한다.
8. 이미지 계획이 승인되면 시스템이 필요한 시각 에셋을 생성하고, 템플릿 도형/텍스트/CTA 레이어와 합성한다.
9. 사용자가 생성물을 최종 검수한다. 수정이 필요하면 이미지 재생성, Filerobot 보정, 레이어 편집, 카피 수정을 거쳐 다시 검수한다.
10. 최종 검수 승인 후에 JPG, 긴 상세페이지 JPG, ZIP, Figma payload, layered document를 export한다.

### B. 기존 상세페이지 리디자인 흐름

1. 사용자가 기존 상세페이지 이미지 또는 PDF를 업로드한다.
2. 시스템이 기존 페이지의 섹션, 카피, 제품 정보, 증빙, 문제점을 추출한다.
3. 시스템이 유지할 정보와 바꿀 정보를 나눈다.
4. 기존 내용을 새 템플릿 기반 레이어 문서로 재구성한다.
5. 사용자는 텍스트와 카드, 이미지, CTA를 직접 조정한다.
6. 필요한 이미지 레이어만 새로 생성하거나 보정한다.
7. 최종 export 전에 기존 정보 왜곡, 과장 표현, 모바일 가독성, 레이어 겹침을 검수한다.

### C. 에디터 내부 흐름

1. 왼쪽 rail에서 섹션을 선택한다.
2. 중앙 캔버스에서 실제 상세페이지 섹션을 본다.
3. 상단 또는 floating workbench에서 작업 모드를 고른다.
4. `이미지` 탭에서는 배경 전체 생성보다 이미지 자산 선택, 교체, 보정, 레이어 삽입을 한다.
5. `레이어` 탭에서는 텍스트, 이미지, 도형, CTA, 카드의 순서와 속성을 수정한다.
6. `카피` 탭에서는 섹션 목적에 맞는 카피 후보를 선택하거나 다시 쓴다.
7. `가이드` 탭에서는 섹션 목표, 근거, 금지 표현, 품질 이슈를 확인한다.
8. 사용자는 Canva처럼 직접 움직이고, Mangoboard처럼 템플릿을 갈아끼우며, 최종 결과를 export한다.

## 이미지 생성의 올바른 위치

현재 정정된 이해 기준으로 이미지 생성은 다음 수준으로 내려가야 한다.

### 좋은 이미지 생성 단위

- 제품 사용 장면 컷
- 제품 단독 누끼 또는 보정 컷
- SW 화면을 담는 노트북/모바일 프레임
- 섹션 배경 패턴
- 비교/증빙 카드에 들어갈 시각 보조 이미지
- FAQ/CTA 섹션의 배경 분위기 이미지
- 기존 제품 이미지를 확장한 배경

### 나쁜 이미지 생성 단위

- 헤드라인과 CTA까지 포함한 완성 섹션 한 장
- 카피 슬롯을 비워둔 거대한 배경 plate
- 모델이 텍스트 배치까지 결정하는 섹션 이미지
- 스펙, 후기, 인증, 가격이 픽셀로 박힌 이미지

## 품질 검수 기준

검수는 두 단계로 나뉘어야 한다.

### 1. 문서 검수

- 텍스트가 캔버스 밖으로 나가지 않는가?
- 버튼과 카드 안에서 텍스트가 잘리는가?
- 레이어가 서로 가리는가?
- CTA가 실제로 보이는가?
- 모바일에서 한 섹션의 정보량이 과하지 않은가?
- 확인되지 않은 수치나 인증이 들어갔는가?

### 2. 이미지 자산 검수

- 제품 또는 SW 화면 정체성이 유지되는가?
- 이미지가 흐리거나 왜곡되지 않았는가?
- 이미지 안에 불필요한 가짜 텍스트가 생성됐는가?
- 원본에 없는 기능, 인증, 리뷰, 수치가 생겼는가?
- 해당 레이어의 목적에 맞는 자산인가?

최종 export 검수는 이 둘을 합쳐 판단한다.

## 앞으로의 구현 방향

### 1. 템플릿을 실제 레이어 프리셋으로 승격

현재 템플릿 레지스트리는 더 강해져야 한다. 각 템플릿은 단순 설명이 아니라 초기 노드 목록을 생성해야 한다.

- headline node
- subheadline node
- product image slot
- supporting image slot
- bullet card nodes
- proof row nodes
- CTA shape + CTA text node
- background shape node

### 2. 이미지 생성 API를 자산 생성 API로 분리

현재 `/api/pdp/images`는 섹션 이미지 생성에 가깝다. 앞으로는 다음처럼 나뉘는 편이 맞다.

- 섹션 전체 생성: 제한적으로만 사용
- 이미지 레이어 생성
- 제품컷 보정
- 배경/패턴 생성
- 기존 이미지 확장
- SW 화면 프레임 생성

### 3. 카피 자동 배치와 텍스트 피팅 강화

텍스트가 레이어로 남으면 자동 배치가 중요해진다.

- 텍스트 길이에 따른 font-size 자동 조정
- 줄바꿈 추천
- CTA 버튼 크기 자동 조정
- 카드 내부 padding 보정
- 모바일 기준 최소 글자 크기 검수

### 4. 최종 문서 기반 export 품질 강화

export 전에는 이미지 생성 품질보다 문서 품질이 더 중요하다.

- 레이어 bounds 검사
- 겹침 검사
- 대비 검사
- OCR 기반 이미지 안 가짜 텍스트 검사
- Figma payload validation
- 긴 상세페이지 연결 시 섹션 간 반복감 검사

## 현재 내가 수정해야 할 이해

내가 이전에 강화한 "배경 plate + 텍스트 안전영역" 관점은 중간 단계로는 유용했지만, 이 프로젝트의 최종 방향으로는 부족하다.

정확한 방향은 다음이다.

이미지가 카피를 위한 공간을 제공하는 것이 아니라, 템플릿과 레이어 문서가 상세페이지 구조를 제공하고 이미지는 그 안에 들어가는 편집 가능한 시각 자산이 되어야 한다.

즉, 생성의 중심은 이미지가 아니라 문서다.
